# pdf-site-log-audit

用于审计工程日志、养护日志、施工日志、巡检日志等 PDF / DOCX 文档，并输出结构化 Word 分析报告的项目。

它适合园林、绿化、市政、养护、施工巡检等台账类文档，尤其适合页面中同时包含：
- 日期
- 天气
- 施工部位
- 作业描述
- 施工照片

## 这个项目是做什么的

这个项目不是单个脚本，而是一套完整工作流。

目标是把原始文档变成可复核、可追踪、可持续批处理的审核流程，最终为每个文档生成一份 `分析报告.docx`。

输出通常包括：
1. OCR / 提取后的文本层
2. 图片层与页面映射
3. 重复图片问题
4. 明确的图文不符问题
5. 最终 `分析报告.docx`

## 可以做什么

这个项目主要解决两类问题：

### 1. 重复图片审核
识别施工照片是否在不同页面被重复使用，并按重复组输出。

### 2. 图文不符审核
结合页面文字和图片，识别明确、严重且无法合理解释的图文矛盾。

项目采用“脚本归一化 + agent 最终判断”的方式：
- 脚本负责 OCR、映射、候选生成、状态持久化
- agent 负责最终复核与判断

## 适用文档

适用于以下类型：
- 工程日志
- 养护日志
- 施工日志
- 巡检日志
- 园林 / 绿化 / 市政项目台账

支持：
- PDF
- DOCX

其中真正关键的分叉是：
- 数字原生文档
- 扫描件 / 图片型文档

## 工作流概览

### 第一阶段：文档归一化

对于数字原生文档：
- 直接提取文本
- 提取嵌入图片
- 建立页面 / 图片映射

对于扫描件：
- 先做 OCR
- 生成文本、图片、layout 等中间层
- 建立 `page-image-text mapping`
- 生成重复图片候选

### 第二阶段：审核判断

#### 重复图片审核
- 脚本先做候选召回
- agent 再做视觉复核与最终归组

#### 图文不符审核
- 脚本先做规则层检查
- agent 再逐页做多模态判断

## 最终报告规则

每个源文档最终生成一份 `分析报告.docx`。

默认只保留两个部分：
1. `重复图片问题`
2. `明确的图文不符问题`

原则：
- 没有问题时写 `未发现...`
- 不写猜测性结论
- 不把抽样结果包装成全量结论
- 同一张重复图片引起的日期/天气矛盾，默认只记在 `重复图片问题`

## 如何使用

这个项目面向 agent 工作流使用。

通常不需要人手动逐个执行所有脚本，而是由 agent 按 skill 说明组织流程、调用脚本、推进批处理，并输出最终报告。

如果要做长任务或批量任务，推荐：
- 后台异步执行
- 结合 OpenClaw cron 续跑

## 部署 / 运行要求

### 1. OCR token
执行 OCR 需要相应 token。

如果当前环境没有可用 token，应前往 Paddle 申请 OCR 服务所需凭据。

凭据不应写入代码，也不应提交到 Git。运行时通过环境变量提供即可。

### 2. Python 环境
项目中的脚本使用 Python 运行。

### 3. Agent 环境
推荐在 OpenClaw / agent 环境中使用，由 agent 统一调度工作流，而不是把它当成单脚本工具直接硬跑。

## 项目定位

这是一个偏 workflow 的项目，不是“一个脚本解决全部问题”的工具。

它的重点是：
- 可复核
- 可持续批处理
- agent 可接管
- 最终结论保留人工/多模态判断层
