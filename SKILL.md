# PDF Site Log Audit

Use this skill for project-log style PDF/DOCX audits where the result must combine script-based normalization with agent-based review.

## Core rule

Do **not** treat this as a script-only task.

This skill has two kinds of work:
1. **Scripted work** for normalization, candidate generation, and progress persistence
2. **Agent work** for duplicate-image confirmation and severe text-image mismatch judgment

The agent must do the final judgment layer.

## Full workflow

### Step 1. Determine document class

Support:
- PDF
- DOCX

Important branch:
- **digital-native document**
- **scanned/image-based document**

### Step 2. Normalize the document

#### If digital-native
Extract directly:
- text
- embedded images
- page/section to image mapping

#### If scanned
Do **not** write "use OCR" abstractly and then improvise.
For scanned PDFs, the agent must follow a concrete script chain with persisted artifacts.

Required path for scanned PDFs:
1. Copy the source PDF into a per-document task directory.
2. Run the OCR extraction script on that local task copy.
3. Persist OCR outputs under the task directory.
4. Build `page-image-text mapping` from OCR output using the mapping script.
5. Generate duplicate-image candidates from the mapped image layer.
6. Only then perform agent duplicate confirmation and severe mismatch review.

The agent must not replace this with ad-hoc browser/model OCR, direct `pdf` tool OCR, or a one-off vision prompt unless the required script path is blocked and that exception is explicitly recorded as a failure/deviation.

Important: the current OCR endpoint only handles up to **100 pages per OCR call** reliably. Split PDFs longer than 100 pages before OCR, then merge all chunk outputs before continuing.

Important: markdown image references are not guaranteed to be complete. Treat the mapping plus exported images as the reliable image layer.

### Required scanned-PDF execution contract

For a scanned PDF, the workflow must explicitly use these artifacts and transitions.

#### A. Task directory
Create one per-document task directory, for example:
- `tmp/<batch_name>/<document_stem>/`

The task directory must contain at least:
- local source PDF copy
- OCR output directory
- rendered/exported image directory
- `page_image_text_mapping.json`
- duplicate candidate artifact
- final review artifact
- final report artifact or pointer to it

#### B. OCR step must be script-driven
For scanned PDFs, OCR must be invoked through the project OCR Python script, not by freeform model prompting.

Required pattern:
- input: local task copy of the scanned PDF
- command form: direct script execution such as `python <script>.py <args>`
- output: persisted OCR files under the task directory

The skill implementation must name and use the concrete OCR script path, for example `scripts/ocr_extract.py`, and the run must leave behind OCR evidence such as:
- `ocr_text/` or `ocr/`
- markdown/text page outputs
- rendered page images and/or exported page-level image artifacts
- raw OCR JSON when available

If these OCR artifacts do not exist after execution, the document must **not** be marked completed.

#### C. Mapping step must be script-driven
After OCR, the agent must run the mapping script, for example `scripts/build_page_mapping.py`, against the task directory artifacts.

Expected output:
- `page_image_text_mapping.json`
- optional mirrored `page_mapping.json`

This file must connect page number, OCR text layer, and image layer. If it is missing, duplicate review and mismatch review are not considered valid.

#### D. Candidate generation must be script-driven
Duplicate-image candidate recall must be generated by script, for example `scripts/find_duplicate_candidates.py`.

Expected persisted output:
- duplicate candidate JSON
- optional contact sheet / candidate montage images

Without candidate artifacts, final duplicate review is not considered fully reproducible.

#### E. Final judgment is agent-only, but must consume script outputs
The agent must make the final duplicate-image and severe mismatch judgment using:
- OCR outputs
- page-image-text mapping
- candidate artifacts
- rendered page images / cropped images

The agent must not skip straight from raw PDF to final report.

### Completion gate for scanned PDFs

A scanned PDF may be marked `completed` only if all of the following are true:
1. task directory exists
2. local PDF copy exists in task directory
3. OCR artifact directory exists and is non-empty
4. `page_image_text_mapping.json` exists
5. duplicate candidate artifact exists
6. final review artifact exists
7. final `分析报告.docx` exists beside the source PDF

If any item is missing, the task must stay in a non-complete state such as:
- `prepared`
- `ocr_done`
- `mapping_done`
- `candidate_done`
- `review_done`
- `failed`

Do not mark such a file `completed` based only on a generated report or a conversational conclusion.

### Required completion validation script

Before a scanned-PDF task is marked `completed`, run:
- `scripts/validate_scanned_completion.py <task_dir> <source_pdf>`

This validator is the required completion gate.
If the validator returns any failed check, the task must not transition to `completed`.

### Cron configuration references

When creating or auditing cron-driven batch execution, use these references:
- `references/cron-message-template.txt`
- `references/cron-template.md`
- `references/cron-add-example.sh`
- `references/execution-contract.md`

## Script responsibilities

Use scripts for deterministic or repeatable steps.

### `scripts/prepare_scanned_pdf.py`
Use for the first deterministic normalization step for scanned PDFs.

Expected responsibilities:
- copy source PDF into task_dir
- render page images
- generate initial cropped image regions
- write `doc_meta.json`
- write an initial `page_image_text_mapping.json`

This script is the required entrypoint before OCR/mapping/candidate generation for scanned PDFs.

### `scripts/ocr_extract.py`
Use for OCR extraction of scanned files.

Required usage rule:
- for scanned PDFs, this is the primary OCR entrypoint
- invoke it as a direct Python script command
- point it at the local task-directory PDF copy, not the original source path outside the task workspace

Expected output:
- markdown/text pages
- exported images
- layout images
- raw OCR JSON
- non-empty OCR artifact directory under the task directory

### `scripts/build_page_mapping.py`
Use for building the stable `page-image-text mapping` from OCR output and logs.

Required usage rule:
- run only after OCR artifact generation succeeds
- write `page_image_text_mapping.json`
- this file is mandatory for scanned-PDF completion

### `scripts/build_page_mapping_local.py`
Use when local OCR/page text artifacts need to be folded back into the canonical mapping file.
This is a helper for the same mapping stage, not a replacement for OCR.

### `scripts/find_duplicate_candidates.py`
Use for duplicate-image candidate generation.
This is only the recall stage, not the final decision.

Required usage rule:
- consume mapped image artifacts from the task directory
- persist candidate JSON and optional contact sheets/montages

### `scripts/find_duplicate_candidates_local.py`
Local helper for generating duplicate candidates from mapped cropped images when running fully from local artifacts.
Persist candidate JSON before agent review.

### `scripts/make_candidate_contacts.py`
Use to render candidate contact sheets/montages for human or agent duplicate review.
This improves reproducibility of the final duplicate decision.

### `scripts/recrop_candidate_regions.py`
Use to re-cut candidate image regions when the first heuristic crop is not suitable for duplicate review.
The recrop step must still feed back into candidate generation and review artifacts.

### `scripts/extract_page_summaries_local.py`
Optional helper for deriving compact per-page textual summaries from OCR text artifacts.
Useful for mismatch review, but not a substitute for full OCR artifacts.

### `scripts/full_review_worker.py`
Use for background batch continuation of page-by-page review.
This script is for progress advancement and persistence, not for replacing final multimodal judgment.

Required usage rule:
- it may advance page review state and persistence
- it must not alone decide the final duplicate-image conclusion or final severe mismatch conclusion without agent judgment

## Agent responsibilities

The agent must handle the parts that cannot be safely reduced to a single deterministic script.

### A. Duplicate-image confirmation

After candidate generation, the agent must:
- review candidate images visually
- decide whether they are real duplicates, near-duplicates, or not duplicates
- group final findings by duplicate-image lineage, not just page pairs

Rules:
- if one page has multiple images and only one repeats across pages, still report that page group
- if different repeated images produce different cross-page groups, report them separately
- if the same repeated image propagates across several pages, merge into one group
- when a duplicated photo is reused across different dates, weather, or text entries, record it under `重复图片问题` only by default
- do not create an additional `明确的图文不符问题` entry when the only contradiction is caused by the same duplicated photo being reused on another page

### B. Severe text-image mismatch review

The agent must review page by page.
For each page, combine:
- extracted text
- page images
- date
- weather
- location
- work description

Only report **clear severe mismatches**.
A mismatch is severe only when it is not reasonably explainable, such as clearly incompatible weather, season, or scene context.
Do not report merely because the pictured work content is not an exact wording match to the text when the difference is still reasonably explainable within the same maintenance context.
If the reason cannot be stated concretely, do not report it as a severe mismatch.
Do not report weak suspicion.
Do not treat sampling as a final-document conclusion.

## Long-running execution model

For long audits, prefer:
- **background async batch execution**
- **OpenClaw cron** for continuation

The correct architecture is:
- persisted status files
- persisted progress logs
- small concrete batches per run
- cron-triggered continuation for the remaining pages

Do not rely on one long blocking foreground turn.

### Cron is the required default for batch rework

For multi-file scanned-PDF rework batches, this skill should normally be driven by a background cron job, not by repeated manual foreground turns.

The expected pattern is:
1. Create a batch task list JSON file
2. Create a cron job that processes exactly one pending file per run
3. Each cron run updates task state and leaves persisted artifacts
4. The cron continues until no pending files remain

The cron message must explicitly instruct the agent to:
- read the batch task file
- select exactly one pending file
- follow the concrete scanned-PDF script chain
- update task status after each stage
- stop after one file

### Required cron setup guidance

When this skill is used for background batch execution, document or invoke cron in a concrete way.
Do not merely say "set a cron".

The skill user/invoker should provide or create:
- a batch task file, e.g. `tmp/<batch_name>_tasks.json`
- one task entry per source PDF
- one task directory per PDF
- a cron job that runs every N minutes and processes exactly one pending task

Recommended job properties:
- session target: `isolated`
- cadence: every 10 minutes by default unless the user asks otherwise
- scope: one file per run
- delivery: optional summary delivery, but must not be required for correctness

### Required batch task schema

Each task entry should include at least:
- `source_pdf`
- `task_dir`
- `status`
- `notes`

Recommended statuses:
- `pending`
- `prepared`
- `ocr_done`
- `mapping_done`
- `candidate_done`
- `review_done`
- `report_done`
- `completed`
- `failed`

Do not collapse all in-progress work into a vague `running` state when stronger stage information is available.

### Required cron completion behavior

A cron-driven file may only move to `completed` after the scanned-PDF completion gate passes.
If a run fails at any point, mark the task `failed` or leave it at the last completed stage with a concrete error note.
Do not leave ambiguous states that make it impossible to tell whether OCR or review truly happened.

## Final output

Generate one final `分析报告.docx` per source document.

By default, the report should contain only:
1. `重复图片问题`
2. `明确的图文不符问题`

Rules:
- write `未发现...` when no issue exists
- do not add speculative sections by default
- do not present sampling as if it were full review
- if a section has no findings, keep only the single `未发现...` row and do not append explanatory prose such as process notes, OCR notes, or why nothing was found
- do not duplicate the same issue across both sections; duplicate-photo reuse belongs in `重复图片问题` unless there is an additional independent contradiction not reducible to photo reuse
- avoid duplicate tables, duplicate rows, flattened one-paragraph output, or mixed title/time formats inside one report
- before overwriting an existing report, compare the newly generated content against the existing report and preserve prior valid findings; fix formatting or wording problems without accidentally deleting established conclusions

### Required pre-overwrite review for existing reports

When a source PDF already has a `分析报告.docx`, do not overwrite it blindly.

Required steps:
1. Read the existing report text first.
2. Compare the existing findings with the newly generated findings.
3. Classify each difference as one of:
   - formatting-only change
   - wording-only change
   - deduplication of duplicated content
   - substantive finding change
4. If the change is substantive, verify it against persisted review artifacts before overwriting.
5. If the goal is only to fix formatting or wording, preserve all prior valid findings and timestamps unless there is a verified reason to change them.

Minimum overwrite safety rule:
- never replace a report containing explicit duplicate-image findings with a new `未发现重复图片问题` conclusion unless the prior finding has been rechecked and explicitly disproved

### Required final report QA checklist

Before considering a report done, verify all of the following:
- exactly one title
- exactly one `分析结果`
- exactly one `分析时间`
- exactly one `一、重复图片问题` section
- exactly one `二、明确的图文不符问题` section
- exactly one 4-column table under each section
- no duplicated rows or stray repeated sentences
- no flattened single-paragraph DOCX output
- no explanatory prose in a `未发现...` section
- no case where duplicated-photo reuse is reported again as `明确的图文不符问题`

## Credential handling

Running OCR requires a valid token.

If the current environment does not already have one, obtain the required credential from Paddle.

Do not hardcode OCR credentials in scripts or documentation examples, and do not commit them to Git. Provide them at runtime through environment variables, but do not document local private config-file details in the public skill instructions.


## Report wording rules

In the final report, do not use wording such as "manual review confirmed".
For duplicate images, prefer neutral wording such as:
- detected duplicate images
- image content is repeated
- duplicate image found

For severe mismatch findings, the reason must be concrete and explain why the contradiction is not reasonably explainable. Do not use vague wording such as "description mismatch" by itself.
